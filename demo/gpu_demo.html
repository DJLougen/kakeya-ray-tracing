Sample Warp (32 threads)

Each cell = 1 thread in a warp. Color = bounce count. Divergent warps stall on slowest. Kakeya sorting groups similar work.

How This Maps to Real NVIDIA Hardware

This WebGL fragment shader executes on the same GPU architecture as NVIDIA RT cores. The warp divergence problem demonstrated here is identical to what occurs in OptiX, Vulkan Ray Tracing, and CUDA path tracers. When threads in a warp hit different code paths (1 bounce vs 8 bounces), the entire warp stalls.

OptiX / RTX Cores

OptiX dispatches rays in warps of 32 threads. Mixed materials cause warp stalls.

Kakeya integration:
Pre-sort rays into SBT indices by predicted workload using polynomial partitioning of the image plane. Feed to optixLaunch().

Vulkan Ray Tracing

vkCmdTraceRaysKHR dispatches shader groups with identical divergence issues. RTX 4090 has 128 SMs x multiple warps each, all vulnerable.

Kakeya integration:
Compute shader pre-pass partitions ray buffer by workload, then reorders SBT before trace.

DLSS Ray Reconstruction

DLSS 3.5 uses ray-traced inputs for temporal upscaling. Uneven ray cost across tiles causes frame pacing jitter.

Kakeya integration:
Partition screen tiles by predicted workload variance, feed balanced samples to the denoiser.

The core insight: Ray tracing is fundamentally a Kakeya problem. Rays are line segments in 3D space. The Kakeya conjecture's polynomial partitioning, dividing space so each cell contains balanced structure, directly applies to balancing ray workloads across GPU warps. This demo shows it running on real hardware. The same math scales to RT cores.
